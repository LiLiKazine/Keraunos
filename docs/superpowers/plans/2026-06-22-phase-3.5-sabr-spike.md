# Phase 3.5 — SABR de-risk spike (detailed plan)

> Spawned from `2026-06-22-keraunos-coverage-roadmap.md` (Phase 3.5). This is a
> **throwaway investigation**, not production code. It **gates Phase 4** (the libav /
> >1080p bundle): the whole libav build is only justified for YouTube if YouTube
> actually serves >1080p over a **fetchable** URL rather than a **SABR-only** stream.

## Question to answer (binary)

For a known >1080p YouTube video, when we force the **web** client + the existing
on-device PoT provider, does the resolved >1080p (VP9/AV1) format come back as a
**fetchable URL** (`URLSession` can GET it) or a **SABR-only** stream (it cannot)?

- **Fetchable → proceed to Phase 4.** libav is justified for YouTube *and* Bilibili AV1.
- **SABR-only → YouTube >1080p is off the table regardless of libav.** Phase 4 then
  justifies itself *only* on Bilibili AV1 — reconsider whether one site is worth the
  permanent libav maintenance cost.

## Why this must run on-device (not in `python-dev`)

The web client requires a **GVS PO token**, and any client requires **nsig**
decryption. Both are minted through the app's JavaScriptCore bridge
(`keraunos_native.eval_js` → `keraunos_youtube_pot.py` / `install_youtube_js_runtime`).
The desktop `python-dev` venv has no `keraunos_native`, so a desktop run fails PoT/nsig
before it can ever observe a format URL. **The spike has no valid localhost proxy — it
must run against live YouTube from inside the running app.**

## The exact pass/fail marker (code-grounded)

yt-dlp's YouTube extractor decides SABR-vs-fetchable at
`app_packages/yt_dlp/extractor/youtube/_video.py:3463-3479`:

```python
fmt_url = fmt.get('url')
if not fmt_url:
    sc = urllib.parse.parse_qs(fmt.get('signatureCipher'))
    fmt_url = url_or_none(try_get(sc, lambda x: x['url'][0]))
    encrypted_sig = try_get(sc, lambda x: x['s'][0])
    if not all((sc, fmt_url, player_url, encrypted_sig)):
        msg = f'Some {client_name} client https formats have been skipped as they are missing a url. '
        if client_name in ('web', 'web_safari'):
            msg += 'YouTube is forcing SABR streaming for this client. '
        ...
        self.report_warning(msg, video_id, only_once=True)
        continue   # <-- the format is DROPPED; it never reaches the format list
```

So the marker is unambiguous:

- **SABR-only:** the format has **no `url`** and **no recoverable `signatureCipher`** →
  yt-dlp logs *"…missing a url. YouTube is forcing SABR streaming for this client."* and
  **skips** it. The >1080p format simply will not appear in the resolved formats.
- **Fetchable:** the format ends up in the list with a non-empty `url` and
  `protocol` ∈ {`https`, `http_dash_segments`}. (`http_dash_segments` still passes our
  `protocol^=http` check, but note the fragment caveat below.)

## Procedure (on-device, temporary edits — do NOT commit)

1. In `keraunos_extract.py::_extract_impl`, temporarily force the web client:
   ```python
   "extractor_args": {"youtube": {"player_client": ["web"]}},
   ```
   (PoT provider stays registered; `_SUPPORTED_CLIENTS` already includes `web`.)
2. Temporarily relax `_FORMAT` so nothing is filtered out — dump everything:
   ```python
   _FORMAT = "bestvideo+bestaudio/best"   # spike only
   ```
   or set `opts["listformats"] = True` and read the table from stderr.
3. Resolve a known >1080p video (1440p/2160p). Capture stderr (the app already pipes
   yt-dlp warnings to stderr — see the PO-token spike instrumentation).
4. Read off the verdict:
   - **SABR warning present + no >1080p VP9/AV1 format in the dump** → SABR-only → FAIL.
   - **>1080p VP9/AV1 format present with a real `url` and `protocol` https/http_dash_segments**
     → fetchable → PASS.
5. Record the result in the local failure log (Phase 3) and revert the two temporary edits.

## Caveats that change the verdict's meaning

- **`http_dash_segments` ≠ single file.** Even a "fetchable" >1080p DASH format may be
  **fragmented** (many `&sq=N` segments). Fragment assembly is **CUT** (roadmap §2), so a
  fragmented-only result is effectively a FAIL for our native single-`URLSession`-GET
  transfer **unless** it exposes a single complete `BaseURL`. When recording PASS, also
  note whether the format is single-file or fragmented.
- **OTF streams** (`FORMAT_STREAM_TYPE_OTF`, `_video.py:3450`) are skipped outright and
  need init-fragment probing — treat as not-fetchable for our purposes.
- The verdict is a **YouTube-server property**, not a yt-dlp property: it can flip when
  YouTube widens the SABR experiment. Re-run the spike if Phase 4 stalls later.

## Output of this spike

A one-line entry in the Phase 3 local log:
`YouTube >1080p web-client: FETCHABLE(single-file|fragmented) | SABR-ONLY` — which
directly sets whether Phase 4 includes YouTube or is Bilibili-AV1-only.
