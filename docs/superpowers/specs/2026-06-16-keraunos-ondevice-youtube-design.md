# Keraunos On-Device YouTube Support — Design

**Status:** Decided
**Date:** 2026-06-16
**Depends on:** `2026-06-16-keraunos-authenticated-extraction-design.md` (in-app cookie login — already implemented; supplies the cookies that pass YouTube's bot check).

## Problem

After the authenticated-extraction milestone, a user can sign into YouTube in-app and
the captured cookies are replayed to yt-dlp. This *works* — it gets past YouTube's
"Sign in to confirm you're not a bot" gate. But extraction then **hangs forever on
"Resolving…"**, because the post-bot-check path reaches YouTube's **`nsig`/signature
challenge**, which yt-dlp solves with a JavaScript runtime. The embedded CPython has
**no `subprocess`/`fork`** (so no Deno/Node/PhantomJS) and falls back to a pure-Python
JS interpreter that is pathologically slow on-device and **not bounded by
`socket_timeout`** (it is CPU-bound, not a socket read).

Two distinct defects follow:

1. **Infinite spinner.** Extraction has no overall wall-clock bound — only per-socket.
   Any non-socket stall hangs the UI with no recovery.
2. **YouTube doesn't work on-device.** Even bounded, YouTube needs (a) `nsig`
   deciphering and (b) increasingly, **PO tokens** (proof-of-origin, generated by
   Google's BotGuard attestation) for the media URLs.

## Goal

Make YouTube downloads work on-device, and guarantee no extraction can ever hang.

iOS ships **`JavaScriptCore.framework`**, which runs JS **in-process** — no
`subprocess`/`fork` required. That single capability is the key that unlocks both
`nsig` and BotGuard. yt-dlp already does the hard part: it extracts a self-contained
JS snippet and only needs its `console.log` output (`extractor/youtube/_video.py`
nsig fallback). We supply a JS runtime; we do not reimplement YouTube.

## Non-Goals / Deferred

- Multiple YouTube accounts, per-download account selection.
- Non-YouTube sites that need a JS runtime (the bridge is reusable, but only YouTube
  is wired this round).
- HLS/ffmpeg merging (still deferred per the embedded-Python constraints).
- Caching minted PO tokens beyond a single app session (revisit if needed).

## Architecture

The whole feature layers onto **one new native capability** — an in-process
JavaScriptCore evaluator exposed to the embedded Python — plus a safety net and the
Python glue that routes YouTube's needs through it.

```
┌───────────────────────────────────────── Swift (app target) ─────────────────────┐
│ PythonExtractor.resolve(url)                                                       │
│   └─ withTimeout(.seconds(45)) { keraunos_python_extract(url, cookieFilePath) }    │  ← Phase A
│                                                                                    │
│ JSContext eval bridge (ObjC, JavaScriptCore.framework)                             │  ← Phase B-core
│   keraunos_js_eval(script, timeoutMs) -> char*                                     │
│     · console.log capture shim · JSContextGroupSetExecutionTimeLimit backstop      │
│     · network/timer/atob/btoa/navigator shims (added as needed for Phase C)        │
└───────────────────────────────────────────────────────────────────────────────────┘
            │ registered at keraunos_python_init as builtin module `keraunos_native`
            ▼
┌───────────────────────────────────── Embedded CPython + yt-dlp ──────────────────┐
│ keraunos_extract.py                                                               │
│   · JavaScriptCoreWrapper.execute(jscode)  ── mirrors PhantomJSwrapper            │  ← Phase B
│       calls keraunos_native.eval_js(...)                                           │
│   · monkeypatch: YoutubeIE nsig → JavaScriptCoreWrapper (pure-Python nsig never    │
│       runs)                                                                        │
│   · KeraunosPoTokenProvider (yt_dlp .../youtube/pot/ framework)                    │  ← Phase C
│       runs BotGuard (bgutils-js) in the JSC bridge → integrity token → PO token    │
└───────────────────────────────────────────────────────────────────────────────────┘
```

### Components

1. **Bounded extraction (Phase A).** A pure `withTimeout` helper in `KeraunosCore`
   plus a new `KeraunosError.timedOut`. `PythonExtractor.resolve` wraps its blocking
   C call so every extraction resolves or fails within a bounded wall-clock time.

2. **JS-eval bridge (Phase B-core).** An ObjC evaluator built on a `JSContext`, with a
   `console.log` capture shim and a hard per-eval backstop via
   `JSContextGroupSetExecutionTimeLimit`. Exposed to Python at interpreter init as a
   builtin module `keraunos_native` with `eval_js(script, timeout_ms)`.

3. **nsig routing (Phase B).** A Python `JavaScriptCoreWrapper` mirroring
   `PhantomJSwrapper.execute(jscode, …)`, calling `keraunos_native.eval_js`. YouTube's
   `nsig` is wired to it so the slow pure-Python path never runs.

4. **PO-token provider (Phase C).** BotGuard attestation JS (vendored `bgutils-js`)
   run inside the *same* JSC bridge with native shims; integrity-token exchange; PO
   token minting; registered as a yt-dlp `PoTokenProvider`.

## Data Flow — gated YouTube extraction

1. `resolve(url)` runs with the user's cookies (from the auth milestone), wrapped in
   `withTimeout`.
2. yt-dlp `YoutubeIE` fetches the player JS. For `nsig`, yt-dlp extracts the
   self-contained function snippet and calls `JavaScriptCoreWrapper.execute(...)` →
   `JSContext` returns the decrypted `n`.
3. When a GVS PO token is required, yt-dlp's pot director calls
   `KeraunosPoTokenProvider`: BotGuard runs in the JSC context, the integrity token is
   exchanged, and a PO token is minted bound to visitor/datasync id.
4. yt-dlp assembles AVFoundation-muxable format URLs (existing `_FORMAT` selection) +
   the PO token → `_payload_for_info` → JSON → Swift downloads natively.

## Phase A — Bounded extraction

- **`withTimeout`** (`KeraunosCore`):
  ```swift
  func withTimeout<T: Sendable>(_ seconds: Duration,
                                _ operation: @escaping @Sendable () async throws -> T) async throws -> T
  ```
  Races `operation` against `Task.sleep(for: seconds)` in a `withThrowingTaskGroup`;
  the first to finish wins, the loser is cancelled, and on expiry it throws
  `KeraunosError.timedOut`.
- **`KeraunosError.timedOut`** — new case with `errorDescription` ("Extraction took too
  long and was stopped."). Swift-originated; not produced by the Python decoder.
- **`PythonExtractor`** — wraps the blocking `keraunos_python_extract` call in
  `withTimeout(.seconds(45))`. The timeout duration is injectable via `init` for tests.
  45s clears any valid extraction; `socket_timeout=15` still catches network stalls
  faster.
- **Limitation (documented):** a synchronous CPython C call holding the GIL has no
  cancellation point, so on timeout the orphaned call keeps running on the serial
  executor until it returns; the next extraction queues behind it. Acceptable for a
  single-user app and rare once Phase B replaces the nsig hang.

## Phase B — JavaScriptCore eval bridge + nsig

- **Native (`PythonRuntime/`, ObjC):**
  `char *keraunos_js_eval(const char *script, double timeoutMs)` — builds a
  `JSContext`, installs a `console.log` shim capturing output into a buffer, evaluates
  `script`, and returns a malloc'd UTF-8 result the caller must `free()`. On JS error
  or timeout it returns an error sentinel string. The per-eval hard backstop uses
  `JSContextGroupSetExecutionTimeLimit` so a pathological script can't wedge the
  context. (A reused long-lived context vs. fresh-per-eval is an implementation
  decision for the plan; Phase C wants a reusable, shimmed context.)
- **Exposure to Python:** during `keraunos_python_init`, register a builtin module
  `keraunos_native` exposing `eval_js(script, timeout_ms)` as a `PyCFunction` that
  wraps `keraunos_js_eval`. No compiled extension module or subprocess required.
- **Python glue (`keraunos_extract.py`):**
  ```python
  class JavaScriptCoreWrapper:
      def __init__(self, extractor, *, timeout=5000): ...
      def execute(self, jscode, video_id=None, *, note='Executing JS') -> str:
          # calls keraunos_native.eval_js(jscode, self.timeout); returns stripped output
  ```
  Wire YouTube's `nsig` to this wrapper so the **pure-Python nsig path never runs**
  (it is the hang). The exact monkeypatch tactic (replace `PhantomJSwrapper`, or wrap
  `_decrypt_nsig` reusing `_extract_n_function_code`) is decided in the plan; the
  invariant is "no pure-Python nsig." To contain monkeypatch drift, **pin the yt-dlp
  version** and add a guard test that the patched symbols still exist.
- **Signature (`sig`) deciphering stays pure-Python** — it is a short, fast function;
  only `nsig` is pathological.
- **Risk:** a bare `JSContext` has no `window`/`document`/`navigator`. If a player's
  nsig body references browser globals, add minimal shims to the context.

## Phase C — PO tokens via BotGuard (research spike)

- Vendor `bgutils-js` (BotGuard runner) into the bundle; run it in the **same** JSC
  bridge with native shims: network (`fetch`/`XMLHttpRequest` → routed through Python
  `urllib` or native `URLSession`), `setTimeout`/`setInterval`, `atob`/`btoa`,
  `TextEncoder`/`TextDecoder`, and a minimal `navigator`/`window`.
- Fetch the BotGuard challenge program → run the VM → exchange for an **integrity
  token** (Google's `GenerateIT` RPC) → **mint GVS PO tokens** bound to visitor data /
  datasync id.
- Implement `KeraunosPoTokenProvider` against yt-dlp's `extractor/youtube/pot/`
  provider framework so the extractor pulls tokens automatically for the relevant
  contexts.
- **Explicit fallback / graceful degradation:** if BotGuard-in-JSC proves infeasible
  (anti-tamper, endpoint churn, missing shims), ship A+B. PO-token-free videos play;
  the rest fail cleanly via the error mapping and **never hang**. The bundle delivers
  value regardless of C's outcome.

## Error Model

- `KeraunosError.timedOut` (new, Swift-originated) — surfaced by Phase A.
- JS-eval failures (syntax/runtime/timeout) map to `.runtime` or `.unsupported` with a
  clear detail.
- A "PO Token Required" outcome from yt-dlp maps to the existing `.requiresAuth` (the
  user can re-check sign-in) or stays `.unsupported` with a clear message — never a
  hang.
- The cookie layer's fail-open behavior (from the auth milestone) is unchanged.

## Testing

- **Core (`swift test`):** `withTimeout` (resolves, times out, propagates errors) and
  `KeraunosError.timedOut.errorDescription`.
- **Swift (`xcodebuild test`):** a `JSContext` eval test (JavaScriptCore is available in
  the test bundle) — verify a JS computation, `console.log` capture, and that the
  execution-time-limit backstop fires.
- **python-dev (`pytest`):** test `JavaScriptCoreWrapper` and the nsig monkeypatch with
  an **injected fake `eval_js` backend** (a Python callable) — validates
  snippet-building and wiring without the real native bridge. Add a guard test that the
  yt-dlp symbols the monkeypatch targets still exist (drift detection).
- **Device/manual acceptance (the true gate):** sign into YouTube in-app, then download
  (a) a Shorts URL and (b) a regular video; confirm they resolve and the `.mp4`
  appears, that a second same-site URL reuses the session, and that the spinner never
  hangs. Capture results in a `docs/logs/` entry.

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| yt-dlp version drift breaks the monkeypatch | Pin the yt-dlp version; guard test asserts patched symbols exist. |
| Incomplete browser-global shims | Start minimal; add shims as specific players/BotGuard demand. |
| BotGuard anti-tamper / endpoint churn (Phase C) | Structured as a spike with explicit fallback to A+B. |
| Added bundle size from vendored JS | Measure `.app` size in acceptance; `bgutils-js` is small relative to the Python stdlib already bundled. |
| Orphaned C call after a Phase A timeout | Documented; rare once B lands; revisit only if it bites. |

## Build Order

Implemented as phases within one plan, layered: **A** (independent safety net) → **B**
(JSC bridge + nsig, the foundation) → **C** (PO tokens, builds on B's bridge). Each
phase leaves the app green and shippable; C degrades gracefully if the spike falls
short.
